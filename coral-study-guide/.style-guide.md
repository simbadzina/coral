# Style guide for the Coral Study Guide

This file is the contract every chapter (and every sub-agent authoring a chapter) follows. Read it before writing or revising any chapter.

## Voice & tone

- **Direct and assertive.** Write what is, not what might be. "`HiveToRelConverter` parses Hive SQL," not "`HiveToRelConverter` may be used to parse Hive SQL."
- **Second person for instructions, third person for descriptions.** "You set a breakpoint at..." vs. "`ParseTreeBuilder` walks the AST."
- **Senior engineer audience.** Assume Java, SQL, and basic Calcite literacy. Don't explain what a class is. Do explain Coral-specific patterns the first time they appear.
- **No marketing.** No "powerful," "robust," "seamless," "elegant." Describe the mechanism.
- **No hedging.** Avoid "essentially," "basically," "kind of," "sort of."

## Structure

- Each chapter opens with a **one-paragraph thesis** (2-4 sentences) — what this chapter is about, what you'll know after reading it.
- Immediately after the thesis, a **front-matter blockquote** with reading time, prerequisites, and 3 key takeaways:

  ```markdown
  > **Reading time** ~14 min  ·  **Prerequisites** [chapter 01](01-the-big-picture.md), [chapter 03](03-pipeline-deep-dive.md)
  >
  > **Key takeaways**
  > - <one declarative sentence grounded in the chapter body>
  > - <…>
  > - <…>
  ```

  Reading time ≈ word count ÷ 150, rounded up. Prerequisites links the chapters to read first; write `Prerequisites none` if there are none. Key takeaways are 3 (occasionally 2-4) single sentences that reflect actual content.
- Then H2 sections. Use H3 for sub-sections inside an H2.
- Each chapter closes in this order: a **`## Self-check`** section (3-4 active-recall questions, no inline answers), then a **"Files this chapter discusses"** list (paths only, no descriptions), then a **"Read next"** pointer to 1-3 related chapters. Reference chapters (glossary, community) may carry a lighter self-check or omit it.
- Stubs: just the H2/H3 headings + 1-2 sentences under each describing intent. No prose body yet.

## Code references

- Always cite specific files with relative paths from repo root, in backticks: [`coral-common/src/main/java/com/linkedin/coral/common/ToRelConverter.java`](../coral-common/src/main/java/com/linkedin/coral/common/ToRelConverter.java).
- When citing a specific line or method, use `file_path:line` or `file_path#methodName` so the reader can jump there.
- Class names in backticks: `HiveToRelConverter`. Method names with parens: `convertSql()`.
- Use code blocks (` ```java ` or ` ```sql `) for any snippet longer than a method signature.

## Diagrams

- **Use Mermaid for all diagrams.** GitHub renders Mermaid natively, so the markdown stays self-contained.
- Prefer `graph LR` for pipelines, `graph TD` for hierarchies, `sequenceDiagram` for time-ordered interactions, `classDiagram` for type relationships.
- Label edges with the class/method that does the transition (e.g., `SqlNode -- ParseTreeBuilder --> RelNode`).
- Keep diagrams under ~15 nodes; split if larger.

## Length

- **README**: ~600 words.
- **Stub chapters**: 150-300 words.
- **Full chapters**: see per-chapter target in the README TOC. Generally 1,500-3,000 words. Quality over filler.

## Conventions

- LinkedIn-internal acronyms: define on first use in each chapter (`Dali (LinkedIn's logical dataset platform)`). The full glossary lives in `19-glossary.md`.
- Recent PRs worth citing: link with `#NNN` and a one-line summary. Don't link to PRs more than 18 months old unless they introduced architecture still in place.
- When referencing source line numbers, treat them as point-in-time — they'll drift. The class name and method name are more durable.

## What not to do

- Don't write "Coral is a powerful library that..." — write what it does.
- Don't repeat content across chapters. Cross-link instead: "see [chapter 07](07-transformers-pattern.md) for the transformer pattern in detail."
- Don't author "TODO" sections in published chapters. If a section is incomplete, the whole chapter stays a stub.
- Don't include emojis or marketing-style flourishes.
- Don't include line numbers in prose unless they're load-bearing — class + method name is enough.
